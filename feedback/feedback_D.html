!关键 Bug — 立即修复

这三个问题会完全阻塞用户或使核心页面无法使用，必须在下一版本修复。

P0 "开始使用" 按钮在勾选协议后无法跳转
📍 入口：法律协议页（首页默认入口）

勾选两个必读复选框后，"开始使用" 按钮的视觉状态看起来正常（无 disabled 灰色），但点击后页面不跳转、Streamlit 也不重渲染。DOM 验证显示 disabled: false，但事件回调未触发路由切换。

建议修复：事件回调可能绑定在了 label 而非 button 元素，或 session_state 键引用了过期变量。直接将 on_change 绑定到 checkbox 状态并强制 st.rerun()。
P0 健康档案页出现未渲染的原始 HTML 字符串
📍 健康档案 → "我的健康状况" 与 "过敏原" 区段标题

页面正文直接显示了 HTML 源码：

<div class='profile-section-title'>我的健康状况</div> <div class='profile-section-desc'>可多选，帮助我们提供更准确的建议</div>

这些本应是带样式的 section 标题，但以纯文本形式泄露在页面上。

建议修复：检查 st.markdown(html, unsafe_allow_html=True) 是否被错误转义，或占位符 {} 字符串未替换成实际 DOM 元素。
P0 首页 "点击大按钮开始" 提示气泡遮挡扫描按钮
📍 首页 → 居中绿色大圆形扫描按钮

橙色提示气泡 点击大按钮开始 直接覆盖在绿色扫描按钮上，文字被截断为 点击大按钮开。在 tablet / mobile 视口下问题更严重，气泡几乎完全隐藏。

建议修复：将气泡定位从 absolute top:0 改为 absolute top:-50px（按钮上方）或使用 CSS pointer-events + 浮动动画配合 transform: translateY(-100%)。

▦整体布局与视觉设计

应用采用了鲜明的品牌色彩体系和卡片化布局，但在信息密度、留白和导航层级上还有改进空间。

✓ 做得好的部分

  • 三色风险系统（绿/黄/红 + 圆/三角/方块 + 文字）兼顾老年人与色盲用户，图例清晰
  • 品牌色统一：绿色 #2E7D32 + 橙色 #FF9800 贯穿全应用
  • 结果页评分 Hero 卡视觉冲击力强，大字号分数 + 可放心食用 pill 极为清晰
  • 装饰性圆角元素（如分数卡角部圆形装饰）增加质感而不嘈杂
  • 卡片化布局节奏良好：分数 → 添加剂 → 健康建议 → 语音

✗ 需要改进

  • 侧边栏占用桌面约 20% 宽度，混合了导航、模型选择、历史记录预览，应拆分为独立区域
  • "选择识别模型" 暴露 Agnes 2.0-flash 等技术信息，给老年用户造成认知负担，应折叠到"高级设置"
  • 首页中间大量留白：扫描按钮居中但内容左对齐，视觉重心失衡
  • Streamlit 浏览器工具栏泄露：右上角 Fork、GitHub 图标、Connecting 状态指示器稀释品牌

↗用户交互流程与易用性

3 步 3 秒的核心 Slogan 深入人心，但进入流程存在阻断、加载态缺失等问题。

流程优势

  • 3 步极简流程（拍照 → 识别 → 听结果）是设计最强资产，全程贯彻
  • "+ 添加健康状况" 胶囊是复杂表单的轻量入口
  • 底部双按钮（再扫一个 / 返回首页）符合老年用户决策习惯
  • 语音播报 + 0.7x/1.0x/1.3x 语速调节对视障/阅读困难用户极其友好

流程缺陷

  • 法律协议门禁过严：必须勾选两个复选框才能进入，且按钮本身有 Bug（见 P0 #1）。建议 MVP 阶段默认勾选，用户主动 opt-out
  • 协议展开后无视觉提示：用户不知道必须先展开才能勾选，应加"请展开阅读"提示
  • 扫描过程无明确 Loading：10-15 秒识别期间仅出现右上角"Stop"按钮，缺少"正在识别"遮罩
  • 缺少相机拍照入口：README 提到支持 st.camera_input，但部署版本只有文件上传，移动端用户体验不完整
  • 未提供相机权限引导：若启用相机调用，需要清晰的"允许使用相机"教程

⊙扫描输入与结果显示

扫描页输入控件有溢出问题；结果页评分卡优秀但缺少营养数据。

输入页（拍照/上传）

  • ✅ 标题 "📷 拍照或上传配料表" 配图标清晰
  • ✅ 拍摄提示 "对准包装上的配料表，保证光线充足、文字清晰"
  • ✅ 文件约束 "支持 jpg / png，最大 5MB" 明确
  • ❌ Upload 按钮溢出卡片右侧边框，文字 "Upload" 与 "MB per f..." 被截断。Streamlit 默认 file_uploader 未做容器宽度适配
  • ❌ "Browse files" 为英文，与全中文 UI 不一致，应自定义 label
  • ❌ 选择文件后布局突变：上传控件移到右侧、预览在左侧的并排布局显得割裂，预览应在下方或替换原控件

结果页（识别结果）

  • ✅ 大号分数 "84" 是视觉焦点
  • ✅ 产品名 + "慢速重听" 按钮在 Hero 卡内即可触发，操作路径短
  • ✅ 添加剂清单 + 色盲图例（圆=安全 三角=中等 方块=高风险）
  • ❌ 缺少营养成分条：测试图片中明确包含营养成分表（能量 2406 千焦、蛋白质 4.4g、脂肪 43.6g、钠 109mg），但结果页未显示任何 NRV% 进度条。README 中提到的 render_nutrition_bars 似乎未生效
  • ❌ "特定人群注意" 标签太笼统：山梨糖醇和甜菊糖苷都标同一标签，应区分 "糖尿病慎用"/"肠胃敏感慎用"/"孕妇儿童慎用"
  • ❌ "查看全部配料" 折叠区被埋：应作为主按钮放在添加剂清单后，因为老年用户最想看完整配料表
  • ❌ "评分仅供参考..." 免责声明出现两次（Hero 卡下方 + 页脚），重复

≡页面元素一致性

字体、颜色、间距在大多数场景下保持一致，但按钮形态、导航组件仍有局部冲突。

字体

  • ✅ 中文 PingFang SC / Noto Sans SC + 数字 Roboto 字体栈合理
  • ❌ 标题字号不统一：首页"食品配料表识别"~24px，法律页"用户协议及隐私政策"~22px，扫描页"扫描识别"~20px，应统一
  • ❌ 侧边栏导航带 emoji 前缀（🏠/📋/👤）但主内容页标题没有，应统一
  • ❌ 中英文标点混用（如文件名省略号 "test_image..."）

颜色

  • ✅ 绿/橙/红三级体系含义清晰
  • ❌ 橙色被过度使用：操作按钮、再扫一个、筛选 pill、添加健康状况、模型选择边框、历史卡片边框全是橙色，应仅用于主 CTA，其他使用中性灰
  • ❌ 浅黄背景的"可放心食用"Hero 与白色卡片容器对比稍弱，可加深边框色

间距与圆角

  • ✅ 16-20px 卡片内边距合理
  • ❌ 按钮圆角不统一：开始使用是全圆胶囊、添加健康状况是胶囊、再扫一个是 4px 直角，应标准化
  • ❌ 首页元素之间的间距不均（扫描按钮上方大空白、下方小空白）

组件复用

  • ❌ 两个"返回"指示器：扫描/结果页用顶部 ← 箭头，首页用"返回首页"按钮，应统一为粘性左上角返回
  • ❌ 两个"健康档案"入口：首页有 "+ 添加健康状况" pill + 顶部 ♥ 心形图标，应只保留一个
  • ❌ 语音控制分裂：结果页既有大号"一键播报全部结果"按钮，又有"语速调整"折叠器，应合并为单一语音面板

▢移动端响应式与性能

3 个视口（桌面 1280、tablet 768、移动 375）均无水平滚动，但布局策略在移动端显得"被动折叠"而非"主动重设计"。

响应式表现

  • ✅ 触摸目标 48-56px，符合适老化标准
  • ✅ 卡片在移动端正确堆叠
  • ✅ 结果页文字在 375px 宽度下仍清晰可读

移动端问题

  • ❌ 顶部标题栏布局不一致：桌面端标题居中、心形在右；移动端标题左对齐、心形换到第二行。移动端像 fallback，不像有意设计
  • ❌ 汉堡菜单 (>>) 太小且灰色，难以发现。但侧边栏隐藏了关键导航（首页/历史/档案/模型/法律），用户必须盲点
  • ❌ 历史记录卡片在移动端单列展示，浪费大量空间，应改为 2 列网格或更紧凑布局
  • ❌ "点击大按钮开始" 提示气泡在移动端完全遮挡扫描按钮上 1/3 区域
  • ❌ 侧边栏历史区域比主内容卡片还宽，移动端横向空间分配失衡

✓优先级总览

按修复优先级排列，P0 阻塞所有用户，P1 影响核心功能，P2 削弱体验，P3 是细节打磨。

★总结

应用拥有清晰的设计哲学（适老化 + 三色风险 + 语音优先），结果页的评分 Hero 卡是真正出色的组件。然而，三个 P0 Bug 集中爆发在入口门（法律页、健康档案、首页 CTA），表明开发流程可能缺少完整的端到端走查测试——尤其是 ?test=1 这种开发者 URL 之外的真实入口路径。

移动端的扫描流程是真正的核心场景，但目前缺少 st.camera_input 拍照支持，这是产品定位（"3 步 3 秒"）与实现之间的最大差距。建议下一版本优先完成 P0 + P1 的修复，并补一个真实用户测试（让一位 65 岁老人独立完成一次扫描），验证完整闭环。
