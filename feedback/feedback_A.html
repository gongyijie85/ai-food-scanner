二、关键问题清单

P0 - 必须立即修复

问题1：健康档案页HTML标签直接显示为文本

所在位置：app.py 第1875-1894行
影响程度：严重
修复成本：极低（5分钟）
现象：健康档案页面中，"我的健康状况"、"过敏原"等标题显示为 <div class='profile-section-title'>我的健康状况</div> 这样的原始HTML代码。
根因：以下4处 st.markdown() 调用缺少 unsafe_allow_html=True 参数：
  • 第1875行：st.markdown("<div class='profile-section-title'>我的健康状况</div>")
  • 第1876行：st.markdown("<div class='profile-section-desc'>可多选...</div>")
  • 第1893行：st.markdown("<div class='profile-section-title'>过敏原</div>")
  • 第1894行：st.markdown("<div class='profile-section-desc'>如有过敏请勾选</div>")
修复方案：给这4处调用都加上 unsafe_allow_html=True。

问题2：config主题色与CSS主题色不一致

所在位置：config.toml 第13行
影响程度：中等
修复成本：极低（2分钟）
现象：进度条等Streamlit原生组件显示蓝色，而整体设计是绿色主题。
根因：config.toml 中 primaryColor = "#1976D2"（蓝色），但 style.css 中主色是 #2E7D32（绿色）。
修复方案：将 config.toml 的 primaryColor 改为 #2E7D32。

P1 - 建议本迭代修复

编号问题描述影响程度修复成本优先级理由
3 导航系统混乱（侧边栏+顶部返回+页面内跳转，三套并存） 高 中 影响用户理解成本，尤其是老年人
4 识别过程缺少分步进度反馈 中 低 用户等待焦虑，不知道还要等多久

问题3详情：导航系统不统一

现状：
  • 侧边栏有：首页、历史、健康档案 三个按钮
  • 顶部导航栏有：返回按钮
  • 页面内还有：各种跳转按钮
问题：三套导航并存，用户容易困惑；侧边栏在移动端默认展开会遮挡内容；返回按钮逻辑不够直观。
建议方案：
  • 移动端：底部Tab栏（首页 / 历史 / 我的），顶部只保留返回按钮
  • 桌面端：保留侧边栏，但简化内容（只放导航+模型选择）
  • 统一顶部导航栏的交互逻辑

问题4详情：识别过程缺少反馈

现状：点击识别后只有一个加载动画，用户不知道当前在做什么。
建议方案：增加三步提示：
  1. "正在上传图片..."
  2. "正在分析配料表..."
  3. "正在计算评分..."
每个步骤配一个小进度条或图标动画，让等待过程有预期。

问题5详情：移动端导航成本高

现状：移动端用户要切换页面，必须先点左上角的汉堡菜单打开侧边栏，再选按钮，至少两步操作。
建议方案：移动端增加底部固定导航栏（3-4个Tab），符合移动端用户习惯，一步到达。

问题6详情：历史记录功能重复

现状：侧边栏有完整的历史记录列表，还有一个单独的"历史"页面，带搜索和筛选。
问题：功能重叠，用户不知道该用哪个；维护两套代码，改一个容易忘另一个。
建议方案：
  • 侧边栏只显示最近3条历史（快捷访问），加一个"查看全部"链接跳到历史页面
  • 完整的搜索、筛选、删除功能都放在历史页面

三、页面布局与视觉设计

做得好的

  1. 绿色主题契合：主色调 #2E7D32（深绿）搭配浅绿背景，与"健康/食品"主题高度契合
  2. 卡片式布局清晰：各功能模块用白色卡片分隔，层次分明
  3. 大圆形扫描按钮醒目：首页的圆形扫描按钮设计很有辨识度，符合适老化要求
  4. 设计系统完整：CSS中定义了完整的变量系统（色彩、字体、间距、圆角、阴影、动效）

需要改进的

  1. 主题色分裂：config是蓝色，CSS是绿色，原生组件与自定义组件风格不统一（P0）
  2. 按钮风格混杂：主按钮绿色渐变圆角、次按钮橙色边框、侧边栏按钮默认样式+图标 — 建议统一样式规范，最多3种按钮类型
  3. 图标风格不统一：emoji、文字图标、SVG 混用，建议统一用 emoji 保持简单

四、交互流程与易用性

做得好的

  1. 引导流程清晰：4步式引导，每步内容简洁，用户快速上手
  2. 核心路径短：首页 → 点大按钮 → 传图 → 看结果，三步完成核心任务
  3. 适老化到位：18pt基础字号、56px最小按钮高度、高对比度配色、语音播报功能
  4. 错误处理完善：API失败有重试、有降级提示，不直接甩技术错误给用户

需要改进的

  1. 导航系统混乱：三套导航并存，用户容易迷路（P1）
  2. 等待缺少反馈：识别过程只有转圈，用户心里没底（P1）
  3. 返回逻辑不清晰：顶部返回按钮有时候回上一页，有时候回首页，逻辑不一致
  4. 缺少快捷操作：结果页没有"再扫一张"的快捷按钮，用户得返回首页重新来

五、扫描功能输入与结果展示

做得好的

  1. 上传区域直观：虚线边框 + 浅绿色背景 + 相机图标，一眼就知道能传图
  2. 评分英雄区醒目：大字号分数 + 背景色编码（绿/黄/红）+ 形状图标（圆/三角/方块）+ 文字标签，多重编码确保老年人也能快速理解
  3. 添加剂列表清晰：左侧色条快速区分等级、形状图标辅助识别、等级标签明确、底部图例说明
  4. 语音播报实用：结果页有语音播报按钮，语速可调，针对浏览器兼容性做了处理

需要改进的

  1. 缺少拍照示例：用户不知道什么样的照片识别率高（P2）— 建议在上传区域加一张"正确拍照示例"缩略图
  2. 上传提示字偏小："支持 jpg/png，最大 5MB" 等提示文字字号偏小
  3. 添加剂列表缺少折叠：如果有十几二十种添加剂，页面会很长 — 建议默认显示前5种，超过部分"查看全部 X 种"展开
  4. 评分含义不够直观：80分具体意味着什么？建议分数旁边加一句简短说明，如"80分 = 添加剂较少，可放心食用"
  5. 缺少重新扫描入口：结果页底部建议加一个"再扫一张"按钮

六、页面元素一致性

字体

  • 达标 中文字体栈完整：PingFang SC → Source Han Sans CN → Noto Sans SC → Microsoft YaHei
  • 达标 字号分级清晰：display / h1 / h2 / h3 / body-lg / body / body-sm / caption
  • 注意 实际渲染中部分Streamlit原生组件字体没有被覆盖，导致偶尔出现字体不一致

颜色

  • 达标 CSS变量定义了完整的色彩系统（主色、状态色、背景色、文字色、边框色）
  • 问题 config.toml 主题色与 CSS 不一致，导致原生组件颜色跑偏（P0）
  • 注意 按钮颜色种类偏多（绿色渐变、橙色边框、纯文字按钮等），建议收敛

间距

  • 达标 有完整的间距变量体系（xs/sm/md/base/lg/xl/2xl/3xl）
  • 注意 实际使用中部分页面间距不够统一，有的卡片间距大有的小，建议严格使用变量

圆角

  • 达标 圆角体系完整（sm/md/lg/xl/full）
  • 注意 不同组件的圆角选择略随意，建议制定更明确的使用规范

七、响应式与移动端表现

做得好的

  1. 有设备检测机制：detect_device_type() 函数区分 mobile/desktop，兼顾 URL 参数、缓存、User-Agent
  2. 三端都有覆盖：CSS媒体查询覆盖了手机（<768px）、平板（768-1023px）、桌面（≥1024px）
  3. 移动端单列布局：手机端单列，适合单手操作
  4. 安全区适配：考虑了底部安全区（--safe-area-bottom）

需要改进的

  1. 代码重复度高：四对函数（render_home_mobile vs render_home_desktop 等），逻辑大同小异，只是布局不同。建议合并成一套，用CSS控制布局差异。
  2. 移动端导航体验差：没有底部Tab栏，切页要开侧边栏（P1）
  3. 平板端适配较粗糙：768-1023px 这个区间，很多布局是直接复用手机或桌面的，没有专门优化
  4. 侧边栏在移动端默认展开：会遮挡主要内容，建议默认收起

八、代码质量观察

优点

  1. 日志系统完善：有 logging 配置，关键路径都有日志
  2. 缓存策略合理：@st.cache_data 和 @st.cache_resource 使用得当
  3. 安全意识到位：API Key 走环境变量/secrets、有 _safe() 函数做 XSS 防护、生产环境关闭 DEBUG
  4. 错误处理健壮：API调用有指数退避重试，区分可重试/不可重试错误
  5. 注释比较充分：关键函数都有 docstring，复杂逻辑有行内注释

改进建议

  1. 单文件过大：app.py 快 2600 行了，建议拆分为 pages/、components/、utils/、api/ 等模块
  2. 移动端/桌面端代码重复：建议合并，用CSS控制布局
  3. 魔法数字较多：部分硬编码的数值（如间距、颜色值）建议统一用变量或常量
  4. 测试覆盖率：只有 test_core.py 测了核心函数，UI 相关的没有测试
