**商业化、正式上线或融资前，应聘请具备广告法 / 食药监管 / 医疗器械与个人信息合规经验的真人顾问复审。**

## 0. 一句话结论

| 维度 | 验证期 Demo 的安全姿态 |
|------|------------------------|
| **产品定位** | 「配料表科普 / 标签解读辅助工具」，**不是**诊断工具、用药指导、食品安全判定或「能不能吃」的权威结论 |
| **文案红线** | 禁止疗效/治愈/绝对安全/替代医生药师；避免「可放心吃」「禁食」「冲突=不能一起吃」等决断式断言 |
| **评分形态** | 分数只能是「参考分 / 配料关注度」；必须显著免责；不得暗示分数=健康或合格 |
| **药物冲突** | 最高风险功能：仅做「可能相关的文献/说明书级科普提示」，**不得**给停药/换药/禁食指令；商业化前评估药械信息服务备案 |
| **广告/运营话术** | 适用《广告法》绝对化用语与疾病治疗功能禁令；应用商店截图、海报、语音播报、分享卡片同受约束 |

## 1. 主要公开依据（研究索引）

以下为公开法规 / 指南 / 实务中常见被引用的条款方向（条文序号以官方最新文本为准，落地时以现行有效文本核对）：

| 领域 | 公开依据（方向） | 与本产品的关联 |
|------|------------------|----------------|
| **广告宣传** | 《广告法》第9条（绝对化用语）、第16–17条（医疗/药品/器械广告限制；非医药广告不得涉及疾病治疗功能、不得使用易混淆医疗用语）、第18–19条（保健食品广告及变相广告） | 应用商店文案、首页 slogan、分享图、语音播报、运营物料 |
| **绝对化用语执法** | 市场监管总局《广告绝对化用语执法指南》（2023） | 「最安全」「100%无风险」「国家级标准解读」等 |
| **食品安全与食品广告** | 《食品安全法》第71、73、78条等：标签/广告不得涉及疾病预防、治疗功能；保健食品不得宣称疾病预防治疗且应声明「不能代替药物」 | 若产品对「某食品能不能治病/养生」做断言则高风险；对**用户扫描结果**的表述也要避免医疗功效口吻 |
| **普通食品功效** | 《食品安全法实施条例》等 + 市场监管总局/中消协消费提示：普通食品不得宣称保健功能、不得涉及疾病预防治疗 | 结果页勿把普通食品包装成「有助降压/增强免疫」等 |
| **互联网药械信息** | 原《互联网药品信息服务管理办法》；2025 年起国家药监局推进《互联网药品医疗器械信息服务备案管理规定》（审批改备案方向） | **提供药品信息、相互作用类信息**的 App/网站可能落入「药品/医疗器械信息服务」监管；正式运营需个案评估是否需备案 |
| **医疗器械软件边界** | 《医疗器械监督管理条例》及移动/独立软件相关指导原则：以**预期用途**是否用于疾病诊断、治疗、预防等医疗目的为核心 | 若产品自我定位为「个体用药决策 / 能否食用的临床判断」，易被质疑向 SaMD（软件即医疗器械）靠拢 |
| **消费者保护 / 反不正当竞争** | 《消费者权益保护法》《反不正当竞争法》：真实、全面，禁止虚假或引人误解宣传 | 过度承诺识别准确率、安全保证等 |
| **个人信息** | 《个人信息保护法》：健康信息属敏感个人信息，需单独同意、最小必要、告知出境等 | 疾病/用药档案；与文案合规并列，但不替代广告/医疗表述红线 |
| **平台常见实践** | 应用商店健康类 App 描述、竞品免责：「不作为医疗建议」「就医前咨询医生」 | 行业通行防御话术，**不能单独消除违法宣传责任** |

> 本仓库既有内部记录见 `LEGAL_REVIEW.md`、`USER_AGREEMENT.md`：当前 Demo 自我定位为科普展示，不构成医疗建议；扩展药品相互作用时需重新评估备案与资质。

## 2. 风险分层：什么最容易踩线

### 2.1 高风险（验证期也应尽量避免）

1. **疾病治疗 / 预防功效断言**  
   例：「降血压食品」「抗癌配料」「增强免疫力，少生病」「吃了能辅助治疗糖尿病」。  
   依据方向：《广告法》第17条；《食品安全法》食品广告不得涉及疾病预防、治疗功能。

2. **诊断 / 处方级指令**  
   例：「您患有××，这款食品确诊不宜食用」「请立即停药」「根据您的病开出饮食方案」。  
   易被理解为诊疗行为或医疗器械软件用途。

3. **药物-食物冲突的绝对禁令**  
   例：「华法林 + 绿叶菜 = 禁止食用」「吃了会中毒/致命」「必须停药」。  
   个体差异、剂量、医嘱优先级极高；错误指令可导致人身损害与产品责任风险。

4. **安全性 / 功效绝对化**  
   例：「100%安全」「绝对无害」「最安全添加剂」「国家级最权威评分」。  
   依据方向：《广告法》第9条 + 《广告绝对化用语执法指南》。

5. **把普通食品/添加剂评分包装成医疗或保健功效**  
   例：高分 = 「对高血压患者有益」；低分 = 「致病风险」。

6. **未经资质的药品信息「专业服务」外观**  
   完整药品说明书库、相互作用数据库若面向公众持续提供，可能触发互联网药械信息服务备案评估；即使验证期免备案争议，**话术仍应按科普而非药师审方呈现**。

### 2.2 中风险（可用但必须弱化断言 + 显著免责）

- 「风险评分 / 安全分」数字本身（易被老人理解为体检分或合格证）  
- 「建议少吃 / 需关注」类建议口吻（应用「可能需关注」「文献中常被讨论」等）  
- 过敏原/疾病档案联动提示（敏感信息 + 个性化「针对您」表述易像诊疗）  
- 语音播报（口语更易说出「能吃/不能吃」；需与 UI 同一套安全词表）  
- 分享海报、家庭转发（二次传播脱离 App 内免责声明上下文）

### 2.3 相对更安全的定位（仍非零风险）

- 配料 OCR + **对照公开标准（如 GB 2760）** 的名称/编码展示  
- 「常见科普资料中，该成分在××人群中被较多讨论」  
- 「请以包装原文、说明书及执业医师/药师意见为准」  
- 「本结果仅供理解标签，不构成能否购买/食用的决定」

## 3. 验证期 Demo：禁止表述 vs 更稳妥替换（话术表）

> 原则：**把「结论」改成「信息 + 边界」；把「你必须」改成「请与专业人士确认」；把「安全」改成「参考/关注度」。**

| 场景 | ❌ 高风险 / 禁止或极不建议 | ✅ 验证期相对更稳妥 |
|------|---------------------------|---------------------|
| **产品 slogan** | 3 秒告诉你能不能吃；智能饮食医生；替代营养师 | 帮你看懂包装上的配料；配料表科普辅助；识别结果仅供参考 |
| **首页/商店描述** | 专治选错食品；老年人饮食安全卫士（暗示保证安全） | 面向配料表阅读困难用户的标签解读演示；非医疗器械 |
| **评分名称** | 安全分、健康分、能吃指数、医疗风险分 | 配料参考分、关注度参考、添加剂分类参考分 |
| **高分文案** | 可以放心吃；完全安全；适合您长期食用 | 根据当前规则暂未标出需特别关注的项；仍请以包装与医嘱为准 |
| **低分文案** | 危险，不能吃；有毒；致病；立即扔掉 | 规则命中若干需关注项；是否食用请结合自身情况咨询医生/药师 |
| **直白结论（老人向）** | 能吃 / 不能吃 / 建议购买 / 不建议购买（决断） | 信息提示：有 N 项值得留意；是否购买/食用请自行或与家人、医生决定 |
| **疾病联动** | 您有糖尿病，此食品禁止；对您的病有疗效 | 档案中含××，部分配料在公开资料中常被该人群讨论；请咨询医生 |
| **过敏原** | 检测到过敏原，食用必死/严重过敏（恐吓式断言） | 识别到与您填写过敏原可能相关的名称，请务必核对包装并遵循医嘱 |
| **保健食品** | 这款保健品能降三高、代替药物 | 保健食品不是药物，不能代替药物治疗疾病；请阅读蓝帽子说明与禁忌 |
| **绝对化** | 最权威、第一、100%、根除风险、永不误判 | 基于公开标准与演示规则；AI 与规则可能有误差 |
| **疗效/医疗用语** | 消炎、抗癌、溶栓、处方级饮食治疗、确诊不宜 | 避免医学疗效词；改用「公开标准收录」「常见科普提及」 |
| **语音播报** | 「这块可以吃」「千万别吃」 | 「识别完成，有若干项供您留意，详情请看屏幕，并请以包装和医生意见为准」 |
| **药物冲突（见第4节）** | 禁止同服；必须停药；会致命 | 公开资料中二者常被一并讨论；用药与饮食调整请咨询医师或药师，勿自行停药 |

### 3.1 评分产品形态建议（验证期）

| 做法 | 说明 |
|------|------|
| **命名去医疗化** | 避免「安全分」「健康分」；优先「参考分」「关注度」 |
| **分数旁固定免责** | 如：「分数仅反映演示规则下的配料关注度，不代表适合您个人，不能代替医生诊断。」 |
| **禁止合格/认证暗示** | 不得出现「国标合格」「检测通过」除非真有检测资质与报告 |
| **颜色语义** | 红/橙/绿可用于「关注优先级」，文案写「需留意 / 一般 / 较少标记」，避免「危险食品 / 安全食品」 |
| **不与疗效挂钩** | 分数高低不得映射「降血压效果好坏」 |

### 3.2 广告与物料同口径

以下与结果页**同一套禁词表**：App Store / 国内应用商店截图与副标题、小红书/视频号脚本、答辩 PPT、TRAE 论坛介绍、家庭分享卡片、TTS 文案。  
《广告法》第19条方向：不得以介绍健康养生知识等形式**变相**发布医疗、药品、器械、保健食品广告——Demo 讲解应停在「怎么读标签」，而非推销功效。

## 4. 药物-食物冲突：额外谨慎点（重点）

### 4.1 为何单独加严

| 因素 | 说明 |
|------|------|
| **人身损害路径短** | 用户可能按提示自行停药、禁食或加餐，直接影响药效与安全 |
| **个体化极强** | 剂量、适应症、联合用药、肾功能、医嘱优先级无法被 Demo 规则覆盖 |
| **监管外观** | 持续提供药品通用名、相互作用、用药禁忌类信息，更接近「药品信息服务」，正式运营需评估备案（2025 年后备案制改革仍要求合规提供） |
| **证据质量** | 网络流传的「葡萄柚不能配他汀」等存在简化甚至错误版本；错误断言责任高 |
| **老人场景** | 目标用户识读与核实能力弱，更依赖「一句话结论」，产品更应避免决断句 |

### 4.2 功能设计红线（验证期）

1. **禁止输出行为指令**：停药、改剂量、换药、禁食某类、必须同服/分服时间表（除非完整引用说明书原文且仍加「以说明书与医嘱为准」——验证期建议直接不做剂量/时程建议）。  
2. **禁止确定性因果**： 「吃了会失效/会出血/会中毒」。改为「公开说明书或常见药学科普中提及可能相互影响」。  
3. **禁止完整「用药方案」外观**：勿做成审方工具、药师工作站 UI。  
4. **数据范围最小化**：验证期可用**少量演示用**冲突条目 + 来源标注；避免宣称「覆盖全部药品」。  
5. **冲突结果默认「弱提示」**：标题用「可能相关的注意点」，不用「药物冲突警告（禁止食用）」。  
6. **双通道免责**：结果区 + 首次开启用药功能时的单独说明（敏感健康信息同意之外，增加「非用药指导」确认更稳妥）。  
7. **紧急情形导流**：明确「急性不适 / 疑似不良反应请急救或就医，不要依赖本工具」。  
8. **不采集处方详情做诊断**：若只需演示，用药字段用「自行填写的通用名关键词」即可，并声明可能匹配错误。

### 4.3 药物冲突专用话术表

| ❌ 避免 | ✅ 替换 |
|--------|--------|
| 检测到严重药物冲突，禁止食用该食品 | 您填写的用药与部分配料在公开资料中常被一并讨论；请咨询医师或药师后决定是否调整饮食 |
| 与××药冲突，吃了会……（断言后果） | 说明书或常见科普可能提示留意××类食物与该药的关系；个体情况以医嘱为准 |
| 请停止服用×× | 请勿根据本工具自行停药或改药；任何调整须由具备资质的专业人员指导 |
| 智能用药安全审查已通过 | 未进行药学审方；本页仅为演示性科普匹配 |
| 依据权威药典保证无相互作用 | 演示数据库有限，存在漏报与误报 |

### 4.4 商业化前的额外动作（非验证期必做清单，供 #26 决策）

- [ ] 评估是否构成互联网药品/医疗器械信息服务 → 按现行规定办理备案等手续

- [ ] 律师审《用户协议》中药物模块专章

## 5. 免责声明与协议检查清单（验证期最低覆盖）

> 免责声明**不能**使违法广告或非法行医/无资质诊疗合法化，但能降低「用户误解产品性质」与民事争议风险。下列为产品与法务文案的**检查项**。

### 5.1 展示位置（产品形态）

| # | 检查项 | 验证期建议 |
|---|--------|------------|
| 1 | 首次进入强制阅读/勾选用户协议 + 隐私政策 | 已有实践方向（见 `pages/legal.py`） |
| 2 | 敏感健康信息（疾病、过敏、用药）**单独同意** | 勾选文案写明用途与出境风险 |
| 3 | 结果页**首屏可见**免责（非仅页脚小字） | 评分旁 + 结论区各一条 |
| 4 | 语音播报含简短免责或播报前提示 | 避免只播「能吃/不能吃」 |
| 5 | 药物冲突模块入口二次提示 | 「非用药指导」 |
| 6 | 保健食品场景固定句 | 「保健食品不是药物，不能代替药物治疗疾病」 |
| 7 | 分享/截图带免责或弱化决断标题 | 防止脱离上下文传播 |
| 8 | 运营物料与应用内口径一致 | 商店页、海报、答辩材料 |

### 5.2 实体内容（用户协议 / 结果页文案应覆盖）

| # | 应明确表达的意思 | 示例方向（可润色，非律师定稿） |
|---|-------------------|--------------------------------|
| A | **服务性质**：科普/标签解读演示，非医疗机构、非药师服务 | 「不提供诊疗、用药指导、食品检验」 |
| B | **不构成**：诊断、治疗建议、能否食用的最终决定、质量合格认定 | 「不构成医疗建议或消费保证」 |
| C | **以何为准**：包装原文、药品说明书、执业医师/药师 | 「请以包装与专业人士意见为准」 |
| D | **AI/规则局限**：错识、漏识、规则过时、匹配错误 | 「识别与匹配可能存在误差」 |
| E | **用户责任**：档案真实性；紧急情况就医；勿自行停药 | 见现有 `USER_AGREEMENT.md` 方向 |
| F | **评分含义**：仅演示规则下的关注度，非健康评估 | 「参考分不代表适合所有人」 |
| G | **药物模块**：非相互作用权威审方；数据库有限 | 「漏报误报可能」 |
| H | **数据与出境**（若适用）：敏感信息、境外处理 | 与隐私政策一致 |
| I | **适用对象**：非急救；老人建议家人协助核对 | 适老化场景特有 |
| J | **禁止依赖导致的后果边界** | 责任限制条款（由律师定稿强度） |

## 6. App / Demo 常见合规实践（摘要）

| 实践 | 说明 |
|------|------|
| **预期用途写「非医疗」** | 商店描述、关于页、协议第一条一致写明非诊断、非器械 |
| **健康类免责标配** | 国内上架健康资讯类应用普遍含「不作为医疗处置依据」——必要但不充分 |
| **避免医疗 UI 隐喻** | 听诊器+处方单+「诊断报告」式结果页增加误认风险 |
| **用户生成/档案** | 疾病用药为敏感个人信息：最小字段、可撤回、会话级存储更利于验证期 |
| **不替食品做功效广告** | 工具介绍自己时，不要宣传「扫了就能保健/治病」 |
| **绝对化与治愈率** | 即使是工具广告，也少用「最」「第一」「零误差」 |
| **备案节点** | 验证期境外 Demo 与正式境内运营、药械信息、医疗器械的分界见 `LEGAL_REVIEW.md`；**话术红线与是否已备案无关——未备案也不能乱说** |

## 8. 验证期「可做 / 不做」总表

| 可做（相对稳妥） | 不做（红线） |
|------------------|--------------|
| OCR + GB 等公开标准对照展示 | 宣称诊断、治疗、预防疾病 |
| 配料关注度参考分 + 显著免责 | 「放心吃 / 禁止吃」二元医疗结论 |
| 过敏原名称匹配提示 + 核对包装 | 断言过敏严重程度或预后 |
| 少量药物-食物「一般性科普」+ 就医导向 | 停药/改药/禁食指令或致命断言 |
| 保健食品「不能代替药物」声明 | 普通食品/保健食品功效与疗效宣传 |
| 明确 AI 误差与数据出境 | 「100% 准确」「最权威」「国家级」 |
| 协议勾选 + 结果页免责 | 仅靠页脚小字支撑「能吃」话术 |

## 9. 残留不确定性（需真人顾问确认）

1. 个性化「针对您的疾病/用药」提示在何种强度下被认定为诊疗或医疗器械软件预期用途。  
2. 仅做药食相互作用科普是否一律触发互联网药械信息服务备案（取决于信息深度、是否持续提供药品信息、运营主体与形态）。  
3. 境外 Streamlit Demo 面向境内用户时的广告法执法与平台规则适用细节。  
4. 分享裂变、语音播报是否被整体评价为广告内容。  
5. 适老化「直白结论」与合规弱表述之间的产品取舍（建议验证期优先合规表述，用版式/语音节奏弥补「不够直白」）。
